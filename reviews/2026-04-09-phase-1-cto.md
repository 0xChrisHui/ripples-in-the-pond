# CTO Review 2026-04-09 — Phase 1

**审查定位**：这不是常规的“代码对不对”检查，而是站在 CTO / 产品技术负责人的视角，判断 Phase 1 这轮投入，是否已经把项目从“能跑的 MVP”推进到了“值得继续扩张的产品底座”。

**审查范围**：
- 架构与规范：`docs/ARCHITECTURE.md`、`docs/CONVENTIONS.md`、`docs/STACK.md`
- Phase 1 计划：`playbook/phase-1/track-a-backend.md`、`playbook/phase-1/track-b-frontend.md`、`playbook/phase-1/track-c-integration.md`
- 当前实现：首页、播放器、mint 流程、个人页、合约、cron、数据库迁移、种子数据、发布脚本
- 审查目标：判断“Phase 1 是否完成”和“下一步该如何进入更专业的产品阶段”

**一句话结论**：
Phase 1 是一版**成功的 MVP**，但还不是一版**可以放心放量、继续堆复杂玩法的产品底座**。它已经证明“用户能体验到作品”，但还没有证明“系统能稳定承受真实用户、真实成本、真实异常和真实协作”。

---

## 一、我认可的部分

### 1. 产品主路已经跑通，不再是技术 demo

这轮最值得肯定的，是项目终于从“点亮链路”进入了“用户能感知价值”的阶段：

- 首页已经从单点测试页升级成了作品入口
- 播放器、登录、mint、个人页形成了基础闭环
- 后端已经有 `tracks`、`mint_events`、`health` 等 Phase 1 级别的数据面
- 真实链上 mint 与前端体验第一次发生了连接

这说明项目已经跨过“只会做页面”和“只会做接口”的阶段，开始具备真正的产品形态。

### 2. A / B / C 三线开发思路是对的

虽然执行上还有集成税，但方向是对的：

- 先把后端数据面和合约能力补齐
- 再把前端体验层做出来
- 最后用 Track C 做真实数据接入与全链路联调

从小团队效率角度看，这是比“所有人围着一个文件改”成熟得多的推进方式。

### 3. 技术选型整体保持克制

我认可这轮继续坚持的几个判断：

- 继续用 Next.js App Router，不引入额外前端框架复杂度
- 继续用 Privy 承接登录，不自己造 auth 轮子
- 继续用 viem / Supabase / Foundry，不把栈扩张成维护负担
- 没有为了“高级感”引入 wagmi、ethers、音频库等黑名单依赖

这保持了项目在 Phase 1 仍然可控，没有因为功能变多就把技术栈带偏。

## 二、我最严厉的判断

### 1. Phase 1 的主要短板已经不是“功能够不够”，而是“控制面够不够硬”

如果 Phase 0 的问题是“系统刚能跑”，那 Phase 1 的核心问题就是：

**你们已经开始拥有用户体验和链上成本，但系统控制面还没有同步成熟。**

具体表现为：

- 铸造经济边界没有被硬编码成系统规则
- 链上成功后的落库与补偿链路还不够可信
- 服务端能力边界仍然主要靠约定而不是机制
- 发布门禁没有达到团队可以信任的程度
- 项目治理文档和真实进度开始脱节

这类问题在 demo 阶段可以忍，但在 MVP 之后会直接变成“继续开发越多，潜在风险越大”。

### 2. 现在最危险的风险，已经从“功能性风险”变成“运营性风险”

如果你们现在直接进入 Phase 2，最可能出现的事故不再是：

- 某个按钮不工作
- 某个页面不好看
- 某个接口字段名有点别扭

而是：

- 用户反复 mint 同一素材，稀缺性和 gas 控制一起失效
- 链上已经成功，但数据库没有记住，系统进入“失忆状态”
- 队列与事件表产生不一致，个人页和真实资产脱钩
- 线上构建或类型检查依赖生成物 / 外网状态，不再可预测
- 后续多个 AI / 多个阶段继续接手时，被过期的 `STATUS.md` 和 `TASKS.md` 带偏

这就是我为什么认为下一步不该直接定义成“做更多功能”，而应该定义成“先把产品底座硬化”。

## 三、必须优先解决的问题

### P0. “同一用户 + 同一素材”的唯一性没有闭环

当前真正被保证的只有“相同 idempotency key 不重复”，而不是“同一用户对同一素材不重复”。

这会带来三个后果：

- 产品语义不稳定：按钮上的“已铸造”只是本地状态，不是系统事实
- 经济边界不稳定：刷新、重试、重复点击都可能生成新请求
- 资产语义不稳定：同一个人可能拿到多份本该唯一的 material

如果产品设定就是“可重复购买”，那应该明确建模并设计 UI。  
如果产品设定是“一次铸造即拥有”，那这个问题必须在进入下一阶段前补齐。

### P0. 链上成功后的数据落地不够幂等，存在“链上成功、应用失忆”的风险

当前 cron 的成功路径是：

1. 链上 mint 成功
2. 更新 `mint_queue`
3. 插入 `mint_events`

这个顺序本身没错，但还缺两个专业级护栏：

- `mint_events` 缺少真正的唯一约束
- 成功后若部分落库失败，没有一个“重复执行也安全”的收敛机制

这意味着只要在“链上成功之后、数据库完全写完之前”出一次异常，系统就可能进入最难处理的灰色区：  
**真实世界已经成功，产品世界却没有记录。**

### P0. 合约权限模型还没有以代码形式强制落地

Phase 0 review 里已经提出过热钱包 / admin 边界问题，但到 Phase 1 结束，这件事仍没有被完全收口：

- 部署脚本里没有显式区分 `admin` 和 `minter`
- 合约构造器仍然把更高权限绑定给部署者
- 元数据 URI 仍然带有明显的 placeholder 气质

这说明“安全模型”目前仍部分停留在约定和人为操作层，不是系统层。

### P1. 个人页只展示结果，不展示过程，体验闭环还差一截

现在的 `/me` 更像“成功铸造记录页”，还不是“我的收藏状态页”。

Phase 1 结束时，用户理应至少看到三类状态：

- 已完成
- 处理中 / 待确认
- 失败 / 暂不可见

没有这层状态机，用户会在最关键的情绪时刻丢失反馈：  
点击了 mint，却不知道系统正在发生什么。

### P1. 真实数据和视觉系统之间已经出现契约漂移

mock 阶段的配色和真实 seed 数据现在不是同一套语言：

- 前端组件期待离散设计 token
- 数据库种子存的是十六进制原始值

这类问题表面上只是“颜色没那么准”，本质上是在说明：  
**设计系统还没有和数据模型达成契约。**

如果继续往后做更多岛屿、更多类型、更多情绪主题，这个裂缝会越来越大。

### P1. 发布门禁不稳定，团队还不能完全相信“通过了就能发”

当前有三个明显信号：

- 类型检查会被 `.next` 生成物污染
- 构建依赖外网字体拉取
- 项目脚本还没有统一的 `verify` / `typecheck` / `test` 入口

这会让团队很容易进入一种危险状态：  
**形式上有验证，心理上却不敢相信验证结果。**

一旦出现这种情况，发布纪律就会开始松动。

### P1. 安全与服务端边界仍有“演示期遗留”

这个项目已经过了可以容忍 demo 习惯的阶段，但当前代码里仍残留一些明显的演示期痕迹：

- 登录后把 token 输出到浏览器 console
- 关键服务端模块没有 `server-only`
- 环境变量没有统一校验层

这些都不是“马上会炸”的问题，但它们会持续提醒你：  
系统还是更像一套成长中的原型，而不是一套受约束的生产系统。

### P2. 项目治理文档开始和真实进度分叉

这是这轮非常值得警惕的点。

现在用户已经认为“Phase 1 整体完成”，代码上也基本支持这个判断；  
但 `STATUS.md`、`TASKS.md`、`LEARNING.md`、`ERRORS.md` 的维护明显滞后。

在单人 + 多 AI 协作模式里，这不是“文档晚点补”那么简单。  
它会直接影响：

- 下一个 agent 的判断
- 下一轮 review 的边界
- 下一步计划的依据
- 项目方法论是否真的落地

## 四、第一轮 review 没完全展开的 CTO 视角

### 1. 这已经不是单纯的工程项目，而是“作品型产品”

你们接下来要警惕一件事：

**功能越来越完整，不代表作品感会自然长出来。**

Ripples in the Pond 不是后台系统，也不是工具站。  
它更接近一个“可参与的音乐作品空间”。

所以 CTO 视角下，后面不能只问：

- 有没有更多页面
- 有没有更多 API
- 有没有更多链上对象

还要问：

- 第一屏是否有情绪密度
- 用户第一次 mint 后有没有记忆点
- 个人页是否像收藏空间，而不是数据清单
- 首页和真实数据是否共享同一套视觉语言

这就是“作品型产品”和“功能型产品”的分水岭。

### 2. 这个系统的关键资产不是代码，而是“项目方愿意为用户代付的预算”

只要还在 sponsored mint 模式下，系统的首要资产就不是 NFT，不是页面，也不是数据表，  
而是：**项目方钱包里的可支配 gas 预算。**

这会重排很多优先级：

- 反滥用不再是“增长之后再做”
- 限流与配额不再是“后台优化”
- 熔断不再是“以后加监控再说”

从商业角度讲，`mint/material` 不是普通 API，  
它是一个**预算消耗入口**。  
只要这个认知没被写进系统设计，后面做再多玩法都容易失衡。

### 3. 你们开始需要“可运维设计”，而不是只有“可开发设计”

Phase 1 以前，重点是“能做出来”。  
Phase 1 之后，重点要升级成“出了问题能不能知道、定位、止损、恢复”。

所以接下来的设计目标应该包含：

- 健康检查
- 队列积压可见性
- 关键写路径的结构化日志
- 钱包余额与失败率告警
- 数据补偿脚本或手动恢复手册

没有这些，团队后面会被自己搭出来的系统反噬。

### 4. 项目方法论本身也要产品化

这个项目难得的地方，是你们不只是写代码，也在试图把“和 AI 一起开发”的方法论沉淀下来。

但从当前状态看，这套方法论还没有完全产品化：

- `LEARNING.md` 没成为真正的概念词典
- `ERRORS.md` 没成为真正的错误博物馆
- `STATUS.md` 与 `TASKS.md` 没形成持续可信的协作入口

如果你们认真看待这套工作方式，那这些文件就不是附属物，而是你们的“操作系统界面”。

## 五、我建议的下一步：不要直接进 Phase 2，先插入 Phase 1.5

### 定位

我建议把下一阶段正式命名为：

**Phase 1.5 — Production Hardening Sprint**

目标不是加更多用户可见功能，而是把 Phase 1 的 MVP 升级成一套可以继续扩张的产品底座。

### 建议优先级

#### 第一组：系统与经济边界（必须先做）

1. 明确 `同一用户 + 同一素材` 的唯一性策略
2. 为 `mint_events` 建唯一约束，并重构成功后落库为幂等收敛流程
3. 明确 `admin` / `minter` 地址分离，并把部署脚本改成显式参数化
4. 为 mint API 加最小限流、队列积压保护、全局开关

#### 第二组：发布与服务端边界（紧随其后）

1. 建统一 `verify` 入口，覆盖 lint / typecheck / build / 合约检查
2. 修正 `.next` 生成物污染类型检查的问题
3. 决定字体方案：真正使用 `next/font`，或彻底改为本地/系统方案
4. 给关键服务端模块补 `server-only`
5. 建最小环境变量校验层

#### 第三组：状态体验与作品完成度（随后跟进）

1. `/me` 与 mint 按钮统一展示 pending / success / failed
2. 首页真实数据与设计 token 对齐
3. 补 loading / error / empty 三类显式状态
4. 把“mint 成功”设计成真正值得分享的瞬间

#### 第四组：协作与方法论收口（不要再拖）

1. 更新 `STATUS.md`，明确 Phase 1 已完成
2. 同步 `TASKS.md`，把 Phase 1 收口并提拔 Phase 1.5
3. 补齐 `LEARNING.md` 与 `ERRORS.md` 的最小维护
4. 在 `JOURNAL.md` 记录“先做 Phase 1.5 再进 Phase 2”的决定

## 六、如果我是 CTO，我会如何判断“现在能不能进 Phase 2”

### 可以进入 Phase 2 的前提

- 铸造唯一性规则明确且落地
- 链上成功后的落库流程可重复执行且不会重复记账
- 合约权限模型与文档承诺一致
- 发布验证结果可信
- 个人页与 mint 状态反馈形成完整闭环
- 项目治理文档回到可信状态

### 还不急着做的内容

- 更多岛屿类型
- 更复杂的社交玩法
- 更多 NFT 层级设计
- AI remix 或更重的创作链路
- 扩张型增长机制

这些都很诱人，但都应该建立在更硬的控制面之上。

## 八、最终判断

我对这个项目的评价，比 Phase 0 结束时更高了。  
因为你们已经证明：这不是一个“停留在概念里”的项目，而是一套真正长出了产品形状的系统。

但我也会更严格。  
因为一旦系统开始承载真实体验和真实成本，评判标准就必须升级。

所以我这轮最核心的建议只有一句：

**不要把下一步定义成“继续做更多功能”，而要定义成“先把 Phase 1 的 MVP 升级成可运营、可发布、可扩张的产品底座，再进入 Phase 2”。**
