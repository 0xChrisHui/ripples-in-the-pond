# CTO Review 2026-04-09 — Phase 0

**审查定位**：这不是“代码有没有明显 bug”的常规 review，而是站在 CTO / 产品技术负责人视角，判断这个项目的 Phase 0 是否已经具备继续投入开发资源的资格。

**审查范围**：
- 架构意图：`docs/ARCHITECTURE.md`
- 规范与栈：`docs/CONVENTIONS.md` / `docs/STACK.md`
- 当前实现：前端入口、认证、数据库、mint API、cron、部署脚本
- 当前目标：Phase 0 跑通闭环，并为 Phase 1 打底

**一句话结论**：
Phase 0 是一个**成功的技术原型**，但还不是一个**可以放心往上堆功能的产品底座**。它已经证明“这条链路能跑”，但还没有证明“这套系统能在真实用户、真实成本、真实异常下稳定运行”。

---

## 一、我认可的部分

### 1. 方向是对的

目前最值得肯定的，不是某一段代码，而是整体方向没有走歪：

- 你们选了 **Privy 登录 + 后端代付 gas + 队列异步铸造** 这条路，用户体验方向是对的
- 你们用 **OP Sepolia** 做测试，和未来 OP Mainnet 的生产环境保持一致，这个判断是成熟的
- 你们没有在 Phase 0 就过度实现一整套复杂系统，而是先打通闭环，这个节奏是对的

从创业和产品推进视角看，Phase 0 的价值就是：**证明这个产品不是幻觉**。这一点，你们已经做到了。

### 2. 架构“形状”总体合理

我认同目前文档里最核心的几条方向：

- API 不等待链上确认，前台即时返回，后台异步处理
- 前端不直接持有链上能力，合约交互集中到服务端
- 数据库按 Phase 递增，不在 MVP 之前背一堆未来包袱
- 先用现成 OZ 合约验证主链路，再在后续阶段做真正的产品级合约

这些都符合“小团队 + 高不确定性 + 强产品探索”的最佳实践。

### 3. 你们已经跨过“只会做页面”的阶段

这是一个很关键的里程碑。

很多项目 Phase 0 只是做出一个像样页面，或者一个假 API。你们这里已经包含：

- 用户登录
- 用户身份到钱包地址的映射
- 写入数据库队列
- cron 消费队列
- 链上实际 mint
- Etherscan 可验证结果

这说明你们不是“在做样子”，而是在做真正的 Web3 系统原型。

## 二、我最严厉的判断

### 1. 当前最大的短板不是“功能不够”，而是“控制平面不够稳”

这是我这轮 review 最核心的判断。

你们现在最危险的地方，不在 UI，不在播放器，不在岛屿效果，不在个人页。  
真正的问题是：

**这是一套项目方代付成本的 minted system，但它的控制平面还没有达到“可以放心烧自己的钱”的成熟度。**

具体说，就是下面这几个问题：

- 谁能 mint，边界不够硬
- 抢到哪条任务，边界不够硬
- 出错后怎么恢复，边界不够硬
- 用户能多频繁调用，边界不够硬
- 系统坏了你怎么知道，边界也不够硬

顶级实践里，这些不是“优化项”，而是 sponsored transaction 系统的基本门槛。

### 2. 当前 Phase 0 的风险等级，已经从“功能性风险”变成“运营性风险”

如果你们现在继续按原计划只往前堆 Phase 1 功能，那未来最可能出现的情况不是“某个按钮坏了”，而是：

- 队列重复处理，多 mint
- 钱包泄露，权限扩大
- 某个用户刷接口，把项目方 gas 烧掉
- 某次异常后，队列状态错乱，人工都看不懂
- 线上出问题，但没有统一健康检查、没有错误分层、没有补偿字段

这类问题一旦出现，修起来比做一个新页面难十倍，而且会直接伤害你对项目的信心。

### 3. 现在不应该直接进入“功能型 Phase 1”

我的建议很明确：

**不要把下一步定义为“进入 Phase 1 开发更多功能”。**

下一步应该定义为：

**Phase 0.5 — Platform Hardening Sprint**

先把系统从“跑通 demo”升级到“可持续推进的产品底座”，再上功能。

## 三、必须先修的底座问题

下面这些不是“以后再说”，而是我认为**进入下一阶段前应该收口的核心问题**。

### P0. cron 抢任务必须原子化

当前实现是：

1. 查一条 `pending`
2. 再把它改成 `minting_onchain`

这个流程在并发下会双抢同一条任务，进而双 mint。

对于一个 sponsored mint 系统，这是平台级问题，不是普通 bug。  
如果只允许先修一个问题，我会先修这个。

### P0. 热钱包权限模型和架构承诺不一致

文档里说：

- 热钱包泄露后，损失应被限制在“最多多 mint 一点没价值 NFT”

但当前部署方案里，deployer 同时拥有更高权限。  
而 cron 又直接拿运营钱包发交易。

这意味着：

- 现在的热钱包泄露风险，可能大于文档承诺的风险
- 你们在“安全模型”上，文档和实现还没真正对齐

这不是细节差异，这是架构一致性问题。

### P1. 失败补偿必须绑定当前 job，而不是“第一条处理中任务”

失败处理如果不绑定当前 job，就会在未来留下隐蔽事故：

- 某个任务失败
- 回滚了另一个任务
- 状态看起来“像恢复了”
- 但实际数据已经脏了

这是最难排查的系统类型之一。  
强烈建议尽快改成“显式绑定当前 job id 的状态机补偿”。

### P1. sponsored mint API 需要最小反滥用护栏

你们当前接口已经开始烧项目自己的钱了。  
所以我建议把下列能力前移，不要等到后面：

- 每用户限频
- 队列积压上限
- 全局开关 / 熔断
- 明确的重复请求语义
- 错误信息分级（用户可见 vs 内部日志）

最佳实践不是“等用户多了再加”，而是**一旦系统开始代付成本，就要开始加**。

### P1. 数据库 schema 要开始承担“系统契约”的职责

现在的 schema 更像“够存数据就行”。  
下一步建议把它提升成“系统状态机的载体”。

我会优先考虑这些方向：

- `status` 不再是自由文本
- 增加 `locked_at` / `processing_started_at` / `completed_at`
- 增加 `last_error`
- 为幂等与查询路径建立更明确的索引
- 明确哪些字段由 API 写，哪些字段只允许 worker/cron 写

数据库不是被动记录板，而是系统控制面的一部分。

### P1. 服务端边界要从“约定”升级为“机制”

现在很多关键边界靠注释提醒：

- 这个文件只能后端用
- 这个 key 不能暴露
- 这个模块前端不能 import

但顶级实践会把这些变成机制，而不是口头约定。

建议尽快补上：

- `server-only`
- 明确的环境变量校验
- 服务端模块统一入口
- 前后端 import 边界检查

## 四、第一轮 review 没有完全展开的 CTO 视角

### 1. 产品与品牌体验：现在还是“技术 demo”，还不是“艺术项目入口”

这个项目不是普通工具站，不是后台管理系统，也不是通用模板网站。  
它的第一印象应该更接近：

- 展览入口
- 有情绪的作品空间
- 可以让人驻足 30 秒的沉浸式入口

而当前首页更多是“主链路点亮页”。

这没有错，Phase 0 就该这样。  
但进入下一轮开发时，你们要有意识地区分：

- “功能是否存在”
- “体验是否配得上这个产品”

Phase 1 如果只补功能，不补气质，项目会显得很“工程对、作品感不够”。

### 2. 单位经济学：代付 gas 的系统，滥用防控优先级必须上升

这是一个非常关键的 CTO 视角。

如果每一次调用都在消耗项目自己的预算，那你们的接口设计就不能只按“请求-响应”思维来做，而要按“成本资产”思维来做。

也就是说，`mint/material` 不是一个普通 API，它是一个**资金消耗入口**。

所以你们要把下面这些问题提前思考：

- 一个用户一天最多能花掉你多少预算？
- 一个恶意脚本 10 分钟能烧掉你多少预算？
- 队列积压到多少时，你选择保护系统而不是继续接单？
- 失败重试的上限是按任务、按用户还是按时间窗口？

只要这个问题没被建模清楚，后面功能越多，财务风险越高。

### 3. 可运维性：现在你们能跑，但还不够“可管理”

Phase 0 验证的是“系统能工作”。  
Phase 0.5 应该验证的是“系统出问题时，你们能不能知道、定位、恢复”。

我建议把“可运维性”当成正式 deliverable，而不是附属工程：

- 健康检查
- 结构化日志
- 错误字段
- 余额告警
- 队列可观察性
- 更可靠的 verify gate

一个项目要长期推进，技术团队首先要让自己**不被系统反噬**。

### 4. 学习系统：你们写了规则，但执行没有完全闭环

这个项目非常难得的一点，是你们明确把“用户学习”写进了工程流程。

但目前实际落地还不完整：

- `LEARNING.md` 没跟上
- `ERRORS.md` 没跟上
- 关键代码复述机制没有真正沉淀为文档

如果这个项目真的要成为“和 AI 一起成长”的项目，那这些不是可有可无的小文档，而是项目方法论的一部分。

我建议把学习机制当成正式产物，而不是礼貌性附带。

## 五、我建议的下一步开发顺序

我不建议直接按照原本“做更多功能”的方式进入下一步。  
我建议拆成三个清晰阶段。

### 第一阶段：Phase 0.5 — Platform Hardening Sprint

目标：

- 把系统从“能跑的 demo”升级成“可继续迭代的产品底座”

优先内容：

1. 原子抢单
2. 失败补偿绑定当前任务
3. 热钱包与权限模型梳理
4. 服务端边界硬化
5. 基础限流 / 熔断
6. 队列表结构补强
7. 统一验证脚本和环境一致性
8. 最小健康检查和日志策略

这个阶段做完后，你们会第一次拥有“可以持续开发而不心虚”的底座。

### 第二阶段：Phase 1 — MVP Experience Sprint

目标：

- 补足用户第一轮可感知价值

建议聚焦：

- 更完整的首页氛围与引导
- 登录态闭环
- 播放器体验
- material mint 的状态反馈
- 个人页基础版
- 从“点亮链路”升级成“能让人体验并记住”

也就是说，Phase 1 不只是加页面，而是让作品第一次“像作品”。

### 第三阶段：Phase 1.5 — Operability & Growth Readiness

目标：

- 为多岛屿、合奏、ScoreNFT 做准备

建议聚焦：

- `mint_events`
- `health`
- `check-balance`
- 队列看板 / 可观测性
- 链上同步基础能力
- 面向未来 Phase 2/3 的控制面准备

只有把这层补上，后续复杂玩法才不会建立在脆弱底座上。

## 六、如果我是 CTO，我会怎么定“现在能不能进下一阶段”

我的标准会很明确：

### 可以进入下一阶段的前提

- 队列处理不会双 mint
- 失败补偿不会误改别的任务
- 热钱包权限模型和文档承诺一致
- sponsored mint API 至少有基础防滥用机制
- 服务端边界有机制保护，不只靠约定
- 验证脚本能稳定跑通

### 还不急着做的事

- 多岛屿复杂视觉
- 合奏系统
- ScoreNFT
- AI remix
- 社区钱包
- 空投玩法

这些都很重要，但都不该先于底座修复。

## 八、最终判断

这个项目现在最可贵的地方，不是“代码已经很好”，而是：

- 架构方向大体正确
- 产品野心明确
- 已经跨过只做页面和伪功能的阶段
- 有文档纪律，也有学习纪律

这说明它有做成作品级产品的潜力。

但潜力不等于底座已经稳。  
我这轮最核心的建议只有一句：

**下一步不要定义成“继续开发 Phase 1”，而要定义成“先把 Phase 0 从 demo 升级成可运营系统，再进入 Phase 1”。**

如果你们接受这个节奏，我对项目后续的成功概率会明显更有信心。
