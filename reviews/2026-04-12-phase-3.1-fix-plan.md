## 综合判断

两份 review 的发现分三层：

1. **并发安全 + 永久数据正确性**（Codex 主要贡献）：原子 claim、mint_events 幂等、metadata 占位域名、demo 底曲降级
2. **代码防御性**（Claude 主要贡献）：缺 catch、类型断言、请求体校验
3. **架构闭环**（两份共识）：owner 投影、链上灾备路径

第 1 层最危险——出一次就是不可逆数据事故。第 2 层影响用户体验但不会永久损坏数据。第 3 层是功能完整性。

## Sprint 1：并发安全 + 永久数据（4 个任务，必须在任何新 mint 之前完成）

### F1. process-score-queue 原子 claim
**来源**：Codex P0
**风险**：并发 cron 重复 mint → 孤儿 NFT + metadata 错位
**改法**：
- 新建 Supabase RPC `claim_score_queue_job()`
  - `SELECT ... FROM score_nft_queue WHERE status IN (...) ORDER BY created_at ASC LIMIT 1 FOR UPDATE SKIP LOCKED`
  - 原子写入 `claimed_at = now()`, `updated_at = now()`
  - 返回被 claim 的一条记录（或 null）
- `route.ts` 改为调 RPC 取任务，不再裸 select
- 所有 status 推进加条件 `WHERE id = ? AND status = ?`（CAS）
**文件**：
- `supabase/migrations/phase-3/015_claim_score_queue_rpc.sql`（新建）
- `app/api/cron/process-score-queue/route.ts`（修改）

### F2. mint_events 幂等写入
**来源**：Codex P1
**风险**：重试写重复 → `.single()` 崩溃 → /score/[tokenId] 变 404
**改法**：
- 给 `mint_events` 加唯一约束 `UNIQUE(score_queue_id)` WHERE score_queue_id IS NOT NULL
- `stepSetTokenUri` 改成 `upsert`（以 `score_queue_id` 为幂等键）
- 查 draft 加 null check，缺失时抛错而不是静默写 null
**文件**：
- `supabase/migrations/phase-3/016_mint_events_unique_queue.sql`（新建）
- `app/api/cron/process-score-queue/steps-chain.ts`（修改）

### F3. metadata external_url 用环境变量
**来源**：Codex P1 + Claude Critical
**风险**：永久 metadata 里写死 `ripples.example`，已铸造的 NFT 无法修正
**改法**：
- 新增环境变量 `NEXT_PUBLIC_APP_URL`（必填）
- `stepUploadMetadata` 用 `${NEXT_PUBLIC_APP_URL}/score/${tokenId}`
- 缺失时 fail fast 不铸造
**文件**：
- `app/api/cron/process-score-queue/steps-upload.ts`（修改）

### F4. 底曲缺失时 fail fast
**来源**：Codex P1 + Claude Critical
**风险**：track 没 arweave_url 时静默降级成 demo → "标题对声音错"的永久 NFT
**改法**：
- `stepUploadMetadata`：`track.arweave_url` 为空时直接 `throw Error`
- `score-source.ts`：去掉 demo fallback，arweave_url 缺失时 decoderUrl 返回空
- demo txId 只保留在 decoder HTML 的离线模式
**文件**：
- `app/api/cron/process-score-queue/steps-upload.ts`（修改）
- `src/data/score-source.ts`（修改）

## Sprint 3：架构闭环（2 个任务，可延后到 Phase 4 前）

### F8. /me 展示 owner 而非 minter
**来源**：Codex P1
**问题**：chain_events 已同步但 /me 仍按 mint 人展示
**改法**：
- 明确语义：/me 展示"我当前持有的"
- 基于 chain_events 创建 `score_nft_owners` 投影视图
- `/api/me/score-nfts` 改读 owner 投影
**决策点**：需要用户确认 /me 语义（minter vs owner），再动手

### F9. /score/[tokenId] 链上灾备路径
**来源**：Codex P2
**问题**：DB 丢数据时直接 404，没有 fallback 到链上
**改法**：
- DB 查不到时 fallback：读 ScoreNFT.tokenURI → fetch Arweave metadata → 提取 events
- 降级展示"可播放但信息不完整"
**注意**：Alchemy Free 10 区块限制不影响 tokenURI 读取（view 函数不走 getLogs）
